# 2026-07-03 (2차) 다중 테스트 메일 + 실사용 리스크 점검

## 1. 다중 테스트 메일 (구현 완료)

- 테스트 주소 입력란이 **여러 주소**를 받음 — 쉼표/세미콜론/공백으로 구분,
  중복은 자동 제거(대소문자 무시), 형식이 틀린 주소는 발송 전에 목록으로 경고.
- 입력한 주소 줄은 `config.json`(`test_emails`)에 저장되어 다음 실행 때 자동 입력.
- 내부적으로 테스트 그룹(`Korea contacts (test)`)을 해당 주소들만으로 재생성 후
  `[TEST]` 캠페인 발송 — 기존 수신자 검증 게이트 동일 적용.

변경: `settings.py`(test_emails 저장), `content.py`(`parse_email_list`),
`service.py`(`send_test`가 리스트 수신), `app.py`(파싱·검증·기억), `README.md`.

## 2. 즉시 수정한 리스크 (구현 완료)

**발송 성공 직후 그룹 갱신 실패가 "발송 실패"로 보이는 문제.**
`schedule_now()` 성공 후 `Korea contacts` 그룹 재생성이 네트워크 오류 등으로
실패하면 에러 팝업이 떠서, 사용자가 발송이 안 된 줄 알고 **전체 재발송**할 수
있었음. 이제 그 단계 실패는 에러가 아니라 "발송은 완료됐으나 그룹 갱신 실패,
다음 달 비교가 부정확할 수 있음" 상태 메시지로 처리.

## 3. 파악된 실사용 리스크와 개선 제안

> **후속**: R1·R2·R3·R5·R6·R7·R8은 같은 날 적용됨 —
> `2026-07-03_risk-fixes-r1-r8.md` 참조. R4·R9·R10은 미적용.

우선순위 순.

### R1. 중복 발송 (심각도: 높음)
- 시나리오: 발송 요청은 MailerLite에 도달했는데 응답 수신 중 네트워크가 끊기면
  앱은 에러를 표시 → 사용자가 재시도 → 같은 뉴스레터가 두 번 나감.
  또, 발송 성공 후에도 체크리스트가 ✓로 남아 있어 몇 번의 클릭만으로 재발송 가능.
- 제안: ① 발송 직전에 같은 제목으로 최근(예: 24시간) 발송된 캠페인이 있는지
  API로 조회해 경고. ② 발송 성공 시 확인 체크박스를 해제하고 "이번 호는 이미
  발송됨(캠페인 ID)" 표시 + 재발송하려면 다시 체크하도록.

### R2. 테스트 후 내용 변경 (높음)
- 시나리오: 테스트 발송으로 ✓를 받은 뒤 Markdown 파일·제목·PDF 링크를 바꿔도
  체크리스트는 그대로 → 검증 안 된 버전이 전체 발송될 수 있음.
- 제안: 제목/MD 파일/PDF 링크가 변경되면 `test_sent`를 자동 리셋해서
  테스트를 다시 요구.

### R3. PDF 링크 오타 (높음)
- 시나리오: PDF URL에 오타가 있어도 그대로 발송 → 전 구독자에게 죽은 링크.
  발송 후에는 수정 불가.
- 제안: 테스트 발송 시(또는 카운트다운 시작 전) PDF URL에 HEAD/GET 요청을 보내
  200 OK인지 확인하고, 아니면 경고.

### R4. Suspicious 플래그 주소가 기본 선택됨 (중간)
- 시나리오: 발송 armed 시 전 행이 체크되므로, 봇/의심 주소도 일일이 해제하지
  않으면 그대로 발송됨. 스팸 신고·발신 도메인 평판 하락으로 이어질 수 있음.
- 제안: armed 시 suspicious 행은 기본 해제(체크는 수동으로만).

### R5. CSV 인코딩 CP949 (중간)
- 시나리오: 한국어 Excel에서 "CSV(쉼표로 분리)"로 저장하면 CP949 인코딩이라
  업로드가 UnicodeDecodeError로 실패하고 메시지도 불친절함.
- 제안: utf-8-sig 실패 시 cp949로 재시도하는 fallback.

### R6. 대량 리스트의 속도·레이트리밋 (중간)
- 현재 구독자 1명당 API 1회 호출(업서트). MailerLite 제한은 분당 120회라
  500명이면 수 분 소요 + 429 재시도 반복. 429 재시도가 재귀 호출이라
  이론상 무한 반복 가능(Retry-After는 존중함).
- 제안: 그룹 단위 bulk import 엔드포인트로 전환(호출 수 수십 배 감소),
  429 재시도에 최대 횟수 제한.

### R7. MailerLite 계정에 구독자 무한 누적 (중간, 비용)
- 그룹 삭제는 구독자를 지우지 않으므로 탈퇴자·테스트 주소가 계정에 계속 남음.
  Free 플랜(구독자 1,000명 한도) 사용 시 한도를 갉아먹음.
- 제안: 비교 다이얼로그의 "Departed" 목록에 "계정에서도 삭제(forget)" 버튼 추가
  (선택적, 되돌릴 수 없으므로 별도 확인).

### R8. `[TEST]` 캠페인 잔재 (낮음)
- 테스트할 때마다 캠페인이 대시보드에 쌓여 지저분해지고, 실캠페인과 혼동 여지.
- 제안: 테스트 발송 성공 후 이전 `[TEST]` 캠페인을 자동 삭제(또는 월 1회 정리).

### R9. 보안: API 키 평문 + SSL 검증 꺼짐 (구조적)
- `config.json`에 실 API 키가 평문 저장 + 현재 `skip_ssl_verify: true`로 운용 중
  → 프록시/중간자에게 키가 노출될 수 있는 조합. 키가 유출되면 전체 구독자
  명단 열람·발송이 가능함.
- 제안: ① IT에서 프록시 루트 CA를 받아 `verify=<CA 경로>`로 교체하고
  skip-SSL을 끔. ② 키 저장을 Windows 자격 증명 관리자(DPAPI, `keyring` 패키지)로
  이전. ③ 키가 외부에 복사된 적이 있다면 MailerLite에서 재발급.

### R10. "Sent to N"이 실제 수신자 수보다 클 수 있음 (낮음)
- CSV에 수신거부자가 남아 있으면 MailerLite가 자동 제외하므로 실제 수신자는
  N보다 적음. 숫자만 보면 오해 여지.
- 제안: 최종 확인 다이얼로그에 "이 중 수신거부 K명은 MailerLite가 자동 제외"
  문구 표시(비교 결과 재활용).
